# IronClaw Local Docker Test Environment

Local Docker environment that mirrors the hosted IronClaw deployment. Builds from source with fast incremental rebuilds (~30s after first build). Uses libsql (no external DB), same entrypoint and env var mapping as production (`ironclaw-worker/`).

## Prerequisites

- Docker with BuildKit (Docker 18.09+, enabled by default in modern Docker)
- A local clone of the [ironclaw](https://github.com/nearai/ironclaw) repo
- `NEARAI_API_KEY` from [cloud.near.ai](https://cloud.near.ai)

## Quick Start

```bash
cd openclaw-nearai-worker/ironclaw-local

# Configure (edit .env with your values)
cp .env.example .env
# Set IRONCLAW_DIR, NEARAI_API_KEY, NEARAI_API_URL in .env

# Build from source (first time ~5min, subsequent ~30s)
./run.sh build

# Start container
./run.sh start
# → IronClaw local: http://localhost:18789?token=<auto-generated>
```

## Configuration (.env)

Create a `.env` file (gitignored) with your settings. All config is read from this file.

```bash
# Path to IronClaw source repo (required for build)
IRONCLAW_DIR=/path/to/ironclaw

# NEAR AI Cloud API Key (required)
NEARAI_API_KEY=sk-...

# NEAR AI Cloud API URL
NEARAI_API_URL=https://private-chat-stg.near.ai/v1
```

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `IRONCLAW_DIR` | **Yes** (for build) | — | Path to the ironclaw source repo (or a worktree) |
| `NEARAI_API_KEY` | **Yes** | — | NEAR AI API key |
| `NEARAI_API_URL` | **Yes** | — | NEAR AI API endpoint |
| `IRONCLAW_LOCAL_PORT` | No | `18789` | Gateway port on host |
| `NEARAI_MODEL` | No | `auto` | LLM model to use |
| `RUST_LOG` | No | `ironclaw=debug` | Log level |

You can point `IRONCLAW_DIR` to a git worktree to test a specific branch:
```bash
IRONCLAW_DIR=/path/to/ironclaw/.claude/worktrees/my-branch
```

## Commands

| Command | Description |
|---------|-------------|
| `./run.sh build` | Build Docker image from IronClaw source (BuildKit cached) |
| `./run.sh start` | Start container, print gateway URL with auth token |
| `./run.sh stop` | Stop and remove container |
| `./run.sh restart` | Stop + start (keeps data volumes) |
| `./run.sh logs` | Tail container logs |
| `./run.sh shell` | Exec bash as `agent` user in running container |
| `./run.sh reset` | Stop + wipe all data volumes (fresh instance) |
| `./run.sh url` | Print the gateway URL with auth token |

## How It Works

### Build

The Dockerfile uses Docker BuildKit named build contexts (`--build-context ironclaw=$IRONCLAW_DIR`) so it works regardless of directory layout. The default build context is this directory (for entrypoint.sh and workspace files). BuildKit cache mounts persist the cargo registry, git dependencies, and compiled target directory across builds — only changed code is recompiled.

Build features: `--no-default-features --features libsql` (embedded SQLite, no external DB needed).

### Runtime

The entrypoint maps env vars to IronClaw config, matching the production `ironclaw-worker/entrypoint.sh`:

- **Database**: libsql at `/home/agent/.ironclaw/ironclaw.db`
- **Gateway**: bound to `0.0.0.0:18789` with token auth
- **NEAR AI**: API key auth via `NEARAI_BASE_URL` (stripped of `/v1`)
- **Secrets**: AES-256-GCM with auto-generated master key (persisted on config volume)
- **CLI/Sandbox**: disabled (no TTY, no Docker-in-Docker)
- **Restart loop**: auto-restarts on crash (up to 10 consecutive failures)

Data persists across restarts via named Docker volumes (`ironclaw-local-config`, `ironclaw-local-workspace`).

### Auth Token

On first `./run.sh start`, a random 64-char hex token is generated and saved to `.env.local` (gitignored). The same token is reused across restarts. Use `./run.sh reset` to generate a new one.

## Testing Scenarios

### Fresh Instance

Simulates a brand-new deployment with no prior state:

```bash
./run.sh reset          # wipe volumes + token
./run.sh start          # clean slate
```

### Code Iteration

Make code changes in the ironclaw source, rebuild, and restart:

```bash
# (edit code in $IRONCLAW_DIR/src/...)
./run.sh build && ./run.sh restart
```

### Test a Branch via Worktree

Point `IRONCLAW_DIR` to a worktree in `.env`:

```bash
IRONCLAW_DIR=/path/to/ironclaw/.claude/worktrees/fix-wasm-secrets
```

Then `./run.sh build && ./run.sh restart`.

### Inspect Container State

```bash
./run.sh shell                              # interactive bash
ls ~/.ironclaw/                             # see config, db, channels
ls ~/.ironclaw/channels/                    # installed WASM channels
```

## Differences from Production

| Aspect | Production (`ironclaw-worker`) | Local (`ironclaw-local`) |
|--------|-------------------------------|--------------------------|
| Binary source | GitHub release download | Built from local source |
| SSH | Enabled via `SSH_PUBKEY` | Disabled by default |
| OAuth callbacks | Configured via `OPENCLAW_DOMAIN` | Not configured |
| Rust toolchain | Installed for agent user | Not installed (build is in Docker stage) |
| Rebuild speed | N/A (uses prebuilt binary) | ~30s with BuildKit cache |

## Troubleshooting

### "IRONCLAW_DIR not set"

Add it to `.env`:

```bash
echo 'IRONCLAW_DIR=/path/to/ironclaw' >> .env
```

### Port conflict

```bash
IRONCLAW_LOCAL_PORT=3001 ./run.sh start
```

### Force full rebuild (cache invalidation)

```bash
docker builder prune --filter type=exec.cachemount
./run.sh build
```

### Container keeps restarting

Check logs for the error:

```bash
./run.sh logs
```

The entrypoint has a restart loop (max 10 failures). If IronClaw crashes repeatedly, the container exits. Check `NEARAI_API_KEY` is valid and the model endpoint is reachable.
